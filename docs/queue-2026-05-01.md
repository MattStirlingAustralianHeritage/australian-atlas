## What changed in the last 24 hours

The headline is that the **Mona Farm bug got fixed once, then fixed eight more times**. Yesterday's reviewer published Mona Farm with a Southern Highlands override and the public Rest Atlas page rendered "Canberra, NSW" because three layers below — publish handler, vertical sync, and ISR cache — all conspired to use the legacy `listings.region` text instead of the override. The fix turned out to be architectural, not local: today's commit `02d715e` introduced a `resolveRegionName(listing)` helper in the portal that walks `region_override_id → region_computed_id → legacy text → null`, plus a per-vertical `/api/revalidate` endpoint and a `recordSyncAndRevalidate` helper that purges ISR cache after every sync write. Rest Atlas verified end-to-end the same evening; Mona Farm now renders "Southern Highlands, NSW" correctly.

Today's main session extended that pattern to the eight remaining verticals (Small Batch, Collection, Craft, Fine Grounds, Field, Corner, Found, Table). An audit ([docs/vertical-sync-audit-2026-04-30.md](docs/vertical-sync-audit-2026-04-30.md)) identified that four of those eight (Craft, Corner, Found, Table) had no `region`/`sub_region` column at all and had been conflating suburb with region in a single column; per editor decision those got an additive `sub_region text` migration, with the read-side switch deferred to a separate later session after backfill. The other four (SBA, Collection, Fine Grounds roasters + cafes, Field) were already on the right schema. All nine commits — eight verticals plus a portal feature branch — landed and were verified by 24 force-syncs (3 listings per vertical × 8 verticals) hitting all three success criteria: `sync_log` resolution_source set, revalidation status `revalidated`, and vertical-side region column populated correctly.

The rollout uncovered four operational gotchas worth their own queue entries. **`echo` adds a trailing newline** when piped into `vercel env add` via stdin, which silently broke the secret comparison until I caught it via a `?secret=...%0A` test against a known-working endpoint; the fix is `printf "%s"`. **GitHub→Vercel auto-deploy was missing the webhook** on 4 of the 8 vertical projects (Field, Corner, Found, Table) — every push for the past 22 days had been silently failing to trigger a deploy. **Field's `package.json` had been missing `stripe` for 11 days** since commit `0a6c1e6` introduced the claim flow without updating the dependency manifest; the fix was a one-line addition (`fix(deps): add missing stripe dependency for claim checkout routes`, commit `3ee2c8e`). And along the way, Rest Atlas's homepage `/api/search` was returning HTTP 500 with "Search unavailable" because the route's select list referenced three columns that had been renamed on the `properties` table months ago — fixed by `d52c20b` in a single commit alongside a UX papercut where the trip planner was rendering the literal error code `"no_region"` instead of the friendly message text.

Phase 4 (backfill of the ~4,750 listings whose vertical-side region text didn't match the portal-resolved name) ran in two attempts. The first **halted on Craft at the 5% failure-rate threshold** after 452 successful updates and 24 consecutive `fetch failed` exceptions starting at listing 453 — pattern consistent with Vercel rate-limiting or a serverless function cold-start cascade on `/api/admin/listings/.../retry-push` after sustained admin POSTs at 200 ms cadence. Per editor direction, the second attempt ran with **1 s cadence + a single retry on `fetch_failed` with 3 s backoff**, smaller verticals first, Craft last (resuming naturally from listing #453 because earlier updates fall into the `already_correct` bucket on re-categorisation). The 1 s cadence resolved the issue cleanly: **4,776 listings updated end-to-end across 9 vertical-iterations with zero failures and zero retries fired**. A post-backfill sweep across all matched listings showed **0 mismatches anywhere on the network** — every published listing whose region resolves on the portal now matches what the vertical stores.

The sweep also surfaced pre-existing data anomalies that aren't bugs from this work but are worth queueing: 108 Fine Grounds roasters listed on the portal have `source_id` values that don't map to any roaster row on the vertical (visible product gap — the vertical has only 68 published roasters but the portal claims 154 active), 127 Craft listings have no resolvable region at all (no override, no computed, no legacy text — these need an editorial pass), and a small handful of orphans elsewhere (10 on Found, 1 on Table) plus 5 vertical-only Table listings that never round-tripped to the portal master.

Three structural follow-ups also surfaced today: (1) the 13 unpolygonised regions still block address-to-region resolution for any listing inside them, (2) Craft/Corner/Found/Table render paths still read `suburb` and the read-side switch to `sub_region` is the natural next-session task now that the data is clean, and (3) the new `sync_log` table is now collecting rich audit data that would make a useful read-only dashboard or weekly report.

## Operational lessons to fold into CLAUDE.md / scripts

- **`echo` adds a trailing newline; use `printf "%s"` when piping into Vercel CLI env-add** — silently broke 8 verticals' REVALIDATION_SECRET until detected by the `?secret=...%0A` test.
- **Sustained admin-route POSTs need ≥1 s cadence between calls.** The first Phase 4 attempt at 200 ms cadence halted Craft at 5 % failure rate after ~452 successful POSTs; the cause was almost certainly Vercel rate-limiting or a serverless function cold-start cascade on `/api/admin/listings/.../retry-push`. The 1 s cadence resolved it cleanly across 4,324 subsequent calls with zero retries fired. Rule of thumb: any backfill or sweep that hits a portal admin route in a tight loop should default to 1 s cadence with a single retry-on-fetch-failed wrapper. The `_phase4_backfill.mjs` script's pattern is a working reference.
- **Supabase preview branches need the original schema in `supabase/migrations/`** — older projects whose tables were created via Studio/CLI come up empty when forked, defeating the purpose. For those, direct-apply with row-count + column-list snapshot is the only paranoia available.
- **GitHub→Vercel auto-deploy can silently disable.** 4 of 8 vertical projects had non-firing webhooks for 22+ days. Worth periodic health-check.
- **AI-co-authored commits introducing new package imports must update `package.json` in the same commit.** Field's missing `stripe` dep silently broke 11 days of pushes.
